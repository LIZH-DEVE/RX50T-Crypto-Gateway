# RX50T 架构总览

## 1. 总体架构

当前 `RX50T` 架构不再沿用原 `AX7020/Zynq` 的异构 SoC 模式，而是裁剪为纯 `PL` 版本。

整体通路为：

```text
UART RX
-> Parser
-> ACL
-> Crypto Bridge
-> AES/SM4 Core
-> UART TX
```

辅助控制路径为：

```text
Parser / Top Control
-> Protocol Check
-> Counters
-> Stats Response
-> UART TX
```

## 2. 各模块关系

### UART RX

作用：

- 接收上位机串口数据
- 将异步串口输入转换成内部同步字节流

输出给：

- `Parser`

### Parser

作用：

- 识别帧头
- 读取长度
- 逐字节输出 payload
- 标记 `frame_done / error`

输出给：

- `ACL`
- 顶层控制逻辑

### ACL

作用：

- 根据关键字进行最小规则匹配
- 决定“透传”还是“阻断”
- 当前支持 `4` 条固定阻断规则

输出给：

- `Crypto Bridge`

### Crypto Bridge

作用：

- 8-bit 到 128-bit 的拼包
- 触发加密核
- 128-bit 到 8-bit 的拆包
- 处理短控制帧绕过

输出给：

- `UART TX`

### AES/SM4 Core

作用：

- 完成固定密钥块加密

当前模式：

- `SM4-128`
- `AES-128`

### UART TX

作用：

- 将处理结果回发到 PC

### Top Control / Counters

作用：

- 识别非法模式选择
- 统计 `total / acl / aes / sm4 / err`
- 生成状态查询响应

## 3. 当前协议设计

### 数据帧

基础格式：

```text
SOF(0x55) + LEN + PAYLOAD
```

### SM4 默认模式

- `LEN = 16`
- `PAYLOAD` 直接作为 `SM4` 明文块

### AES 显式模式

- `LEN = 17`
- 第 1 个 payload 字节为 `0x41 ('A')`
- 后 16 字节为 `AES` 明文块

### 显式 SM4 模式

- `LEN = 17`
- 第 1 个 payload 字节为 `0x53 ('S')`
- 后 16 字节为 `SM4` 明文块

### 阻断与错误

- ACL 命中：返回 `D\\n`
- 协议错误：返回 `E\\n`

### 状态查询

- 查询命令：`55 01 3F`
- 响应格式：`53 total acl aes sm4 err 0A`

## 4. 当前模块实现思路

### Parser 思路

- 采用轻量 FSM
- 只做当前纯 PL 主线所需的最小解析
- 不追求完整网络协议栈

### ACL 思路

- 当前实现 `4` 条固定规则
- 目标是在不增加复杂握手的前提下，把单条规则 demo 提升为小型硬件规则引擎

### Crypto Bridge 思路

- 不做复杂控制协议
- 不做动态密钥管理
- 不做硬件 padding
- 先聚焦单块数据的纯计算链

### AES/SM4 思路

- 使用固定测试密钥
- 先完成已知向量闭环
- 保证算法核接入稳定后，再考虑扩展能力

### 状态计数思路

- 计数器全部放在顶层轻量控制逻辑中
- 不污染核心加密桥和 UART 时序
- 通过一条最小查询命令回读，交给上位机做展示

## 5. 为什么这样裁剪

当前裁剪原则是：

- 避免引入 `ARM/PS`
- 避免引入 `DMA/DDR`
- 避免过早追求复杂协议栈
- 用最小系统先证明：
  - 输入链可用
  - 规则链可用
  - 加密链可用
  - 输出链可用

这样更适合 `RX50T` 资源条件，也更适合比赛要求。

## 6. 当前阶段结论

当前系统已经不只是“模块存在”，而是形成了真实闭环：

- 串口输入有效
- 解析有效
- ACL 有效
- AES/SM4 有效
- 串口输出有效
- 状态计数回读有效

因此，当前 `RX50T` 版本已经可以作为：

- 竞赛版主线架构
- 技术文档主体
- 上板演示版本
